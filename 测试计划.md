# 测试计划 (Test Plan)

## 1. 测试综述与目标

本文档详细界定了多主机实时终端系统的测试执行阶段划分与具体验收标准。鉴于系统包含大量底层的 Go Gateway 协议解析（如 `pkg/codec` 帧拆解）及复杂的高频流式渲染，本计划强制要求遵循“自底向上，协议先行”的验证路径。

## 2. 阶段里程碑与进度排期 (总周期：4周)

### 2.1 阶段一：协议底层与 Gateway 接口单元测试 (W1)
- **核心任务**：
  - WSS 握手与 Token/鉴权参数解析（Redis 会话生成）。
  - `gateway/pkg/codec` 模块的健壮性：序列化、反序列化 0x01~0x05 帧。构造 `ErrBufferTooSmall` 和 `ErrInvalidFrame` 边界。
  - FD 限制条件（`ulimit` 启动预检分支）与 Goroutine 内存泄漏验证。
- **准入准出**：
  - 准入：Gateway Router & Session 模块单测跑通。
  - 准出：能够使用构造工具成功与 Gateway 建立长连，互发 `FrameTypeHeartbeat (0x01)` 并维持不掉线。

### 2.2 阶段二：Session 管理与 SSH 链路联调 (W2)
- **核心任务**：
  - `FrameTypeTerminalInput (0x03)` 到 SSH PTY 的写数据链路。
  - SSH PTY 到 `FrameTypeTerminalOutput (0x02)` 的数据组装与多路分发。
  - `FrameTypeWindowResize (0x04)` 参数（Cols, Rows）到 `stty` 的转换。
  - **熔断测试 (isSlow)**：模拟远端不断吐数据，触发 Gateway 写缓冲达到 85% 警戒水位，验证反压分支。
- **准入准出**：
  - 准出：能进行标准单主机会话操作，并能够触发一次成功的 `FrameTypeSessionControl` (如 `SlowWarning` 下发)。

### 2.3 阶段三：Multiexec 分发、防误机制与 UI 端到端 (W3)
- **核心任务**：
  - 在前端（UI）勾选多个机器并发送广播，检验 `SessionCount` > 1 时的帧构造与下发分拆。
  - 智能防误预检与探针分支：针对在 `vim` / `top` 等 TUI 界面中的主机拦截输入广播。
  - Diff Engine (输出差异引擎)：输出哈希对比与异常高亮。
- **准入准出**：
  - 准入：前端 Web Worker 二进制层实现完毕，可联调。
  - 准出：20 台主机的 Multiexec 端到端命令广播保序、无错乱。风险拦截规则通过（含高危命令如 `rm -rf` 阻断）。

### 2.4 阶段四：极限压测、容灾演练与审计验收 (W4)
- **核心任务**：
  - 启动至少 2000 个活跃的 SSH 隧道（单个 Gateway 节点），执行高频日志 `tail -f` 压测。
  - 注入断网故障：触发客户端 15 秒重连分支（会话复用）。
  - 审计功能验收：命令输入及全量输出（包括不可见的控制字符）已序列化入 S3 及 Redis。
- **准入准出**：
  - 准出：无 OOM 问题，FD 句柄完全释放，系统可支撑无中断优雅重启 (Graceful Restart)。所有 Bug 收敛。

## 3. 资源约束与高风险规避

| 风险模块 | 细节描述及触发条件 | 规避与验证方案 |
| :--- | :--- | :--- |
| **内存溢出 (OOM)** | 在极端 `FrameTypeTerminalOutput` 环境下频繁分配 `[]byte` 缓冲引起 GC STW。 | 强制在压测环节通过 pprof 观测 `sync.Pool` 的重用率，不允许存在独立泄漏切片。 |
| **消息乱序 / 数据错位** | Multiexec 下包含不可见控制符 (如 ANSI 色彩序列 `\x1b`) 导致前端终端渲染错乱。 | 对所有命令的 `Input` 分支与 `Output` 分发通道采用强制 FIFO 队列，禁用无序 Map 迭代。 |
| **僵尸连接 / FD 泄漏** | 网络半关闭 (Half-Close) 下未能正确捕获结束信号，导致底层的 SSH Socket 句柄未关闭。 | W4 强制执行 Chaos 拔网线演练，监控 Gateway 所在宿主机 `lsof` FD 数量，确认 15 秒后彻底释放。 |

## 4. 人力及协作要求
本项目因为涉及复杂的底层网络与内存分配，传统 UI 测试工具不足，需安排 1 名专门掌握 Go 和网络协议的“自动化测试开发工程师”专门负责构造 `codec` 畸形帧及底层并发长连接测试。
