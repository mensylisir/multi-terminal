# 测试用例 (Test Cases)

本文档极其详细地描述了架构核心能力、底层二进制帧协议及异常分支条件的测试用例。执行时须监控 Gateway 的资源（Goroutine/FD）与 Web Worker 的协议响应。

---

## 模块 1：底层二进制帧解析与通信 (Codec & WSS)

### TC-FRM-001：正常 FrameType 0x01 (Heartbeat) 的解析与维持
- **API/操作层**：前端建立 WSS 连接后，由 Web Worker 发起。
- **参数与数据格式**：发送 `Type: 0x01, SessionCount: 0, Sessions: []`。
- **触发分支/条件**：系统解析判定为 `FrameTypeHeartbeat`。
- **预期结果**：Gateway 接收并更新当前连接的 `LastActiveAt`，不返回错误；维持 TCP 存活。
- **优先级**：P0

### TC-FRM-002：畸形包与边界条件 (ErrBufferTooSmall / ErrInvalidFrame)
- **API/操作层**：测试工具主动向 Gateway WSS 端点发送异常二进制流。
- **参数与数据格式**：
  - Case 1: 发送总长度为 1 字节的数据 `[0x02]`。
  - Case 2: `SessionCount` 标明为 2，但实际 payload 中只包含了 1 个 SessionBlock。
  - Case 3: 某一个 `SessionBlock` 的 `Length` 标为 5000，但实际后续 Data 只有 10 字节。
- **触发分支/条件**：进入 `codec.Deserialize` 并触发长度越界判断条件 (`len(data) < 2` 或 `offset+int(length) > len(data)`)。
- **预期结果**：底层返回 `ErrBufferTooSmall`，网关在 Router 层打出 Warning 日志，丢弃该包但**绝不能导致 Gateway Panic**，且合法会话应保持不受影响。
- **优先级**：P0

## 模块 2：终端交互与窗口控制 (Terminal I/O & Resize)

### TC-IO-001：单机命令执行与 PTY 交互 (0x03 & 0x02)
- **API/操作层**：在 UI 焦点终端内输入字符 `l`。
- **参数与数据格式**：
  - 上行 (Input)：发送 `FrameTypeTerminalInput (0x03)`，载荷 Data `[0x6C]` (即 'l')。
- **触发分支/条件**：Gateway 解析后投递至目标主机 SSH PTY 的 `Write` 通道。
- **预期结果**：目标机回显，Gateway 下发 `FrameTypeTerminalOutput (0x02)`，Data 为 `[0x6C]`，UI 端成功渲染出 `l`。
- **优先级**：P0

### TC-IO-002：FrameType 0x04 (WindowResize) 触发与系统参数重置
- **API/操作层**：在 UI 端全屏终端，或拖动窗口边缘。
- **参数与数据格式**：发送 `FrameTypeWindowResize (0x04)`，Data 载荷包含编码后的 `{Cols: 120, Rows: 40}`。
- **触发分支/条件**：SessionManager 收到事件并调用 `ssh.Session.WindowChange(40, 120)`。
- **预期结果**：终端下执行 `stty size` 应准确输出 `40 120`。
- **优先级**：P1

## 模块 3：Multiexec 分发与慢节点熔断反压 (Multiexec & Backpressure)

### TC-MUL-001：复杂命令集的无乱序保序广播 (FIFO)
- **API/操作层**：前端在 10 台机器的多选状态下，快速粘贴包含控制符（如换行符、转义符）的命令。
- **参数与数据格式**：上行 `FrameTypeTerminalInput (0x03)`，`SessionCount: 10`，载荷 Data 包含复杂的 bash 脚本及 `\n`。
- **触发分支/条件**：Multiexec Engine 分发至 10 个独立的 Channel 队列中。
- **预期结果**：10 台主机全部成功执行该脚本，无一发生字符截断或因 Channel 拥堵导致的指令重叠乱序。

### TC-MUL-002：缓冲积压与反压熔断 (isSlow = true)
- **API/操作层**：在普通会话中执行 `cat /dev/urandom` 或巨大的日志文件。
- **参数与数据格式**：远端 SSH 不断向 Gateway 灌入海量 `0x02` Output 数据。
- **触发分支/条件**：
  1. 测试人员在前端通过 Chrome DevTools 将网络模拟为 `Slow 3G`。
  2. Gateway 的对应 `Session WriteBufferSize` 不断增长并触发 85% 上限阈值。
- **预期结果**：
  1. Gateway 判断触发条件并执行反压：标志位 `isSlow = true`。
  2. Gateway 停止读取该 Session 的 PTY。
  3. 下发 `FrameTypeSessionControl (0x05)` 告知前端，UI 对该终端标黄提示。
  4. 当网络限速恢复，积压的 Buffer 逐渐耗尽，标志位恢复 `isSlow = false`，恢复 PTY 读取。
  5. 整个过程不得出现内存 OOM 崩溃。

## 模块 4：风控拦截与防误执行 (Risk Control & Consistent Context)

### TC-RSK-001：探针与 PromptState 上下文防误拦截
- **API/操作层**：针对正在处于交互式 TUI 界面（如 `vim` / `top`）的目标机发送广播指令。
- **参数与数据格式**：
  1. 系统在真正执行 `0x03` 输入前，先发送隐藏探针序列 `\x05`。
  2. 目标机的 `PromptState` 鉴别算法匹配探针返回情况。
- **触发分支/条件**：如果匹配出非标准 Bash/Zsh 提示符，判定系统不在命令行外壳中。
- **预期结果**：
  1. Gateway 阻止用户的后续 `0x03` 载荷流入该主机（防止在 `vim` 里疯狂敲代码）。
  2. 前端 UI 通过 `FrameTypeSessionControl (0x05)` 收到拦截通知，将该主机的边框变为红色并触发警示语。

### TC-RSK-002：危险指令 (正则引擎) 的直接拦截
- **API/操作层**：用户在单机或多机终端输入 `rm -rf /usr/lib` 并敲击回车 (`\r`)。
- **参数与数据格式**：正则引擎扫描上行的 `FrameTypeTerminalInput` 行缓冲。
- **触发分支/条件**：匹配到黑名单正则规则（命中高危命令）。
- **预期结果**：不将 `\r` 指令下发给 SSH PTY，直接截断操作，并向前端返回 `FrameTypeTerminalOutput` 打出红色的警告文本 `"Action Blocked by Security Policy"`。

## 模块 5：系统级限制与可靠性 (FD & Connection Handoff)

### TC-SYS-001：Gateway 最大连接与 FD 限额阻断
- **API/操作层**：使用并发压力器模拟创建极大的数量 WSS 连接。
- **参数与数据格式**：模拟 5000+ 的握手请求。
- **触发分支/条件**：Gateway 动态检测系统当前使用的文件描述符（FD）。当 `FD_Usage >= Max_Threshold` 时。
- **预期结果**：触发限流阈值，新的握手请求直接 HTTP 503 拒绝或 WebSocket Upgrade 失败，保障已有连接不受影响。

### TC-SYS-002：会话短时掉线恢复 (Session Resume)
- **API/操作层**：用户主动断开网络 5 秒。
- **触发分支/条件**：Gateway 探测到 WSS TCP 断开，开启 15 秒倒计时但**保留底层 SSH Session 和 Goroutine**。
- **预期结果**：用户网络恢复并在 15 秒内重新 WSS 握手（携带原 Token 和 SessionID），Gateway 迅速绑定旧 Session 句柄，前端页面无感恢复画面。
- **优先级**：P0
