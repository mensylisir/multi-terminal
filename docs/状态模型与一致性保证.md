# 状态模型与一致性保证

本篇定义了系统的核心状态机，涵盖单会话的生命周期管理、客户端的联动状态、以及在批量广播时的远端主机上下文一致性保障。

## 1. 核心状态模型

### 1.1 Session（会话）数据模型
Session 代表了一个逻辑上的终端连接，它跨越了前端的视图与后端的 SSH 隧道。

```go
type Session struct {
    SessionId       uint32    // 帧协议路由的核心标识
    UserId          string    // 归属的鉴权用户
    HostId          string    // 绑定的目标资产标识
    SSHClient       *ssh.Client // 底层网络连接句柄
    ShellStream     io.ReadWriteCloser // PTY 的数据流
    Cols            int       // 当前终端列数
    Rows            int       // 当前终端行数
    State           int       // 状态枚举：active, detached, closed
    LastActiveAt    time.Time // 最后活跃/心跳时间
    WriteBufferSize int       // 衡量发送积压程度的水位线
    IsSlow          bool      // 是否触发慢节点限流熔断
    PromptState     int       // 上下文状态：shell, tui, unknown
}
```

### 1.2 Session 生命周期 (State Machine)
- **`active` (活跃)**: WebSocket 链路畅通，UI 挂载中，正常收发数据。
- **`detached` (游离/断线挂起)**: 当用户页面刷新、网络闪断，或浏览器休眠导致 WS 断开。此时后端的 SSH 并不立刻断开，而是维持其 PTY 及环境变量，暂存最近输出（等待断点重连）。
- **`expired` (过期)**: 处于 `detached` 状态的时间超过系统设定的 TTL（例如 10 分钟）。
- **`closed` (已关闭)**: 触发最终的清理逻辑，销毁 goroutine，发送 TCP FIN，彻底释放对象句柄。

### 1.3 客户端 (Client) 联动状态
```json
{
  "activeSessionId": 1001,
  "selectedSessionIds": [1001, 1002, 1005],
  "mode": "multi" // 可选: single (独立操作) / multi (广播控制)
}
```

## 2. 上下文状态一致性探测 (Context Check)

在执行 Multiexec 广播时，如果某些主机正处于 vim 编辑文本状态，而有些在正常的 bash 环境，贸然输入 `rm -rf` 会导致灾难性且不可预料的按键错位。因此必须执行探测。

### 2.1 隐蔽探测机制
- **探测实施**: Gateway 向所有准备广播的目标 Session 发送一个 `\x05` (ENQ) 字符或不可见的空回车。
- **响应判定**:
  - `shell`: 截获返回输出流中包含了系统配置的 Prompt 特征码（如 `$`、`#`，或者通过 `export PS1` 预先埋点的特定字符串）。
  - `tui`: 返回的流中包含重写屏幕的 ANSI 序列（如清屏、光标跃迁），意味着系统处于 Top/Vim 等全屏占用应用内。
  - `unknown`: 发送探测后，超出 2 秒无任何字符返回，可能由于服务器卡死或网络高延迟。

### 2.2 防护机制触发
系统根据识别出的 `PromptState` 自动做一致性抹平决策：
- 只有所有处于 `shell` 状态的 Session，才允许接受广播输入。
- 对于 `tui` 和 `unknown` 状态的节点，**强行踢出**当前的 Multiexec 广播组（即暂时忽略它们）。
- 在前端 UI 层，给这部分异常终端加上**红色半透明遮罩**，展示“环境异常锁定”警告，在物理和视觉层面双重禁止非法输入的意外写入。
