# 技术选型

本文档详尽说明了支撑本多主机终端系统背后的技术栈选择及其深层考量。

## 1. 后端技术栈 (Gateway)

### 1.1 核心语言: Go (Golang)
- **选型理由**:
  - **原生并发与低开销**: Go 语言提供极其轻量级的 Goroutine 和通信机制 Channel。相较于 Node.js 的单线程事件循环（处理大量二进制 CPU 密集型解析时易阻塞）或 Java 的重型操作系统级线程，Go 能够以极低的内存开销完成并发调度，非常适合处理成千上万个 SSH 连接。
  - **网络性能**: Go 标准库的 `net` 包底层基于 epoll/kqueue 封装，性能优异，在处理长连接、高吞吐和反压流控方面有天然优势。
  - **运维友好**: 编译成单一静态二进制文件，无复杂的运行环境依赖（无需 JVM 或 node_modules），非常契合本系统所强调的易部署、低环境要求和优雅重启特性。

### 1.2 SSH 协议栈: `golang.org/x/crypto/ssh`
- **选型理由**: Go 官方维护的扩展库。它完整实现了 SSH 的客户端行为，包括公私钥鉴权、HostKey 校验、Window Size 控制及 PTY 伪终端的申请，能直接深入到协议层进行定制改造。

### 1.3 WebSocket 通信: `github.com/gorilla/websocket`
- **选型理由**: 这是目前 Go 社区中最成熟、稳定且经过大规模生产验证的 WebSocket 库。
- **匹配特性**: 完美支持自定义二进制 Frame，支持连接层面的精细控制（如 Ping/Pong Timeout 读写截止时间），并能够方便地实现连接升级（Upgrade）。

### 1.4 内存复用机制: `sync.Pool`
- **选型理由**: 在高频输入输出流中，不断创建和销毁切片（Slice）会导致严重的内存垃圾积累和 GC 停顿。通过 `sync.Pool`，我们可以建立不同规格的 Buffer 资源池，请求时借出，用完归还，将 GC 压力降至最低。

## 2. 前端技术栈

### 2.1 终端渲染: `xterm.js` 及其 WebGL 插件
- **选型理由**:
  - 它是目前 Web 终端领域的事实标准（VS Code 等均使用）。
  - 支持完整的 ANSI 解析，具备完备的鼠标、全屏（TUI）控制响应。
  - **核心强制**: 配合 `xterm-addon-webgl` 插件，将原本基于 DOM/Canvas 的渲染卸载到 GPU 处理。在 Multiexec 场景下，可能有 20 个终端并发且满屏高速滚动日志。传统的 DOM 渲染会导致浏览器主线程 100% 阻塞甚至崩溃，而 WebGL 能确保极高吞吐量下的帧率（FPS）稳定。

### 2.2 状态管理: `Zustand` / `Pinia`
- **选型理由**:
  - 轻量、响应式且无需复杂的 Boilerplate 代码。
  - 非常适合管理多维度的终端状态组（比如 `activeSessionId`, 选定的 `selectedSessionIds`，及终端是处于 `shell` 还是 `tui` 等深层嵌套结构）。
  - **防重绘优化**: 在高频更新的终端场景下，如果使用传统的 Context API，任何微小的状态变更都会导致整个多终端网格（Grid）组件树的全量 Re-render。Zustand/Pinia 支持细粒度的属性订阅（Selector），确保仅当某个特定终端的挂载状态（如 `isSlow`）改变时，才触发那个独立终端外壳组件的重绘，而丝毫不干扰内部 xterm 实例的连续渲染。

### 2.3 多线程解析: `Web Worker`
- **选型理由**: 主界面的流畅性（FPS）对于良好的用户体验至关重要。
- 解析从网关传来的高频、聚合二进制帧需要大量的 CPU 时钟和位运算。将这些工作下放到 `Web Worker`，能使得数据的处理和 UI 的渲染实现物理层面的异步解耦。

## 3. 审计存储层

### 3.1 压缩: Zstandard (Zstd)
- **选型理由**: 相较于 Gzip，Zstd 在具有极快压缩和解压速度的同时提供了更高的压缩率。非常适合用来实时流式压缩巨大的 Raw Output 审计日志。

### 3.2 存储方案: S3 / 分片对象存储
- **选型理由**: 审计日志存在“写多读少、体量巨大、结构不可变”的特点，廉价、扩展性极强的对象存储是最完美的归宿。
