# 一、产品目标

在浏览器中提供多主机实时终端能力，并支持安全可控的批量命令执行（Multiexec），同时具备审计与风险防护能力

---

# 二、核心能力

* 多主机 Terminal（每台独立）
* 多选主机 + 输入广播（Multiexec）
* 输出隔离 + 差异识别
* 会话恢复（断线重连）
* 审计（命令 + 回放）
* 风险控制（确认 / 限制 / 审批）

---

# 三、系统架构

```text
Browser（xterm.js + WebWorker）
  │
  │ WebSocket（二进制，多路复用）
  ▼
Gateway（Go）
  ├── Session Manager
  ├── SSH Manager
  ├── Multiexec Engine
  ├── Stream Router（20ms聚合）
  ├── Diff Engine（ANSI Strip + Hash）
  ├── Audit Service（命令 + 流）
  ├── Risk Control
  ├── Resize Controller
  ├── Flow Control（Backpressure）
  ├── Context Checker
  └── Resource Guard
  │
  ▼
SSH Hosts
```

# 五、能力实现

---

## 1. 多主机 Terminal

### 机制

* 每个主机 → 独立 SSH + PTY
* 每个 Session → 对应一个 xterm 实例
* 输入输出按 sessionId 路由

---

### 资源控制

* 单用户最大 Session（建议 ≤ 20）
* 全局最大 SSH 连接数（按机器配置）
* 启动时强制检查：

```text
ulimit -n ≥ 4096
```

---

---

## 2. Multiexec（输入广播）

---

### 执行模型

```text
activeSession → 输入来源
selectedSessions → 执行目标
```

---

### 写入机制

* 每个 session 独立队列
* 异步写入
* FIFO 保序

---

### 慢节点处理

```text
buffer持续增长 → 标记 isSlow
→ 暂停写入该节点
→ UI提示
→ 不影响其他节点
```

---

---

### 上下文一致性检查

---

#### 探测

```text
发送 \x05 或 \n
```

---

#### 状态识别

```text
shell → 包含 prompt（$/#）
tui → 全屏控制字符
unknown → 无响应
```

---

#### 行为

* tui/unknown：
* 红色遮罩
* 禁止输入
* 默认排除

---

---

### 前端回显竞争（新增关键细节）

#### 问题

多节点同时回显 → 字符闪烁

---

#### 解决（必须实现）

在 multiexec 输入时：

```text
1. 前端本地“乐观回显”（立即显示输入）
2. 所有 terminal 临时锁定回显渲染
3. 等服务器输出后统一刷新
```

或：

```text
统一禁用远端回显（短窗口内）
```

## 3. 输出隔离 + 差异识别

---

## Stream Router

---

### 聚合发送

```text
20ms窗口聚合所有 session 输出
→ 单一 WS 帧发送
```

---

### 二进制帧结构（严格规范）

```text
[FrameType:1B][SessionCount:1B]
[
  [SessionId:4B][Length:2B][Data...]
]
```

---

### 字节序（关键约束）

```text
统一使用 Big-Endian（网络字节序）
```

---

### 前端解析要求

* 使用 DataView / Uint8Array
* 严格按偏移读取
* 不允许 JSON fallback

---

---

## Diff Engine

---

### 数据处理流程（固定顺序）

```text
Raw Output
→ ANSI Strip
→ Regex Filter
→ 行切分
→ Hash
→ 对比
```

---

### ANSI Strip（必须）

去除：

```text
颜色 / 光标移动 / 清屏
```

---

### 判定标准

```text
1. 行差异
2. Exit Code（最终裁决）
```

---

---

## 4. 会话恢复

---

### 生命周期

```text
active → detached → expired → closed
```

---

### 机制

* WS断开 → session保留
* TTL（10分钟）
* 重连恢复

---

### 清理

```text
定时器清理 detached
SSH keepalive 检测断链
```

---

---

### 优雅重启（必须实现）

---

#### 流程

```text
1. 广播“维护中”
2. 拒绝新连接
3. session元数据写入Redis
4. 关闭进程
5. 新进程恢复
```

---

## 5. 审计

---

## 命令审计

---

### 输入处理规则

```text
退格 → 删除
Tab → [TAB]
Ctrl+C → [CTRL+C]
Ctrl+Z → [CTRL+Z]
Enter → 提交
```

---

### 输出格式设计 (结构化日志)

为了便于后期的 ELK 或其他日志系统检索，我们采用结构化的 JSON 记录用户的具体操作命令：

```json
{
  "timestamp": "2023-10-25T14:30:00Z",
  "userId": "admin_001",
  "sessionId": 1001,
  "hostId": "web-server-01",
  "commandType": "shell_input",
  "rawInput": "rm -rf /var/log/nginx\r",
  "cleanCommand": "rm -rf /var/log/nginx",
  "riskLevel": "high"
}
```

## 回放审计

---

### 存储内容

```text
Raw Input + Raw Output + Timestamp
```

---

### 存储策略（新增关键细节）

* 回放日志：原始流（便于还原）

---

### 敏感信息处理


# 六、关键机制

---

## PTY Resize

```text
前端 cols/rows → SSH window-change
```

---

## 流控（Backpressure）

---

### 后端

* buffer + 批量发送

---

### 前端

* scrollback限制（建议5000）
* 渲染节流

---

### 保护

```text
输出过快 → 暂停该 session
```

---

---

## WebSocket 优化

---

### 必须

* Binary Frame
* 多路复用
* 批量发送

---

### 建议实现

* 使用 Web Worker 解析二进制帧
* 主线程只做渲染
