# 工程设计

本文档阐述系统整体架构层面的工程化设计，明确前后端各模块的职责、系统边界、线程安全以及通信机制的划分。

## 1. 前端架构与模块划分

前端系统致力于提供极度流畅、可扩展和独立的多终端视图管理能力。

- **终端渲染引擎 (Terminal Component)**:
  - 核心基于 `xterm.js`，强制开启 WebGL 渲染加速扩展以提供最高性能的终端体验。
  - 每个连接对应一个独立的 xterm 实例，由 React/Vue 等框架进行生命周期托管。
- **状态管理层**:
  - 采用现代状态管理库（如 Zustand 或 Pinia）来维护整个应用的数据流向。
  - 管理核心数据结构，如 `ClientState` (当前聚焦 Session、多选目标) 以及 `Session` 的实时状态（活跃、卡顿、断线）。
- **通信解析与多线程计算层 (Web Worker)**:
  - 为防止繁重的二进制解析阻塞 UI 主线程，网络收发与帧解析全部放入 `Web Worker`。
  - Worker 接收 WebSocket 二进制流，通过 `DataView/Uint8Array` 拆包为独立 Session 的输入流。
  - 主线程与 Worker 之间通过高频的 `postMessage` (或 SharedArrayBuffer) 机制进行消息分发。
  - **核心原则**: UI 线程专注于 DOM 和 Canvas/WebGL 绘制，Worker 处理复杂的协议与流逻辑。

## 2. 后端架构 (Gateway)

后端采用 **Go 语言** 编写。基于 Go 强大的 CSP 并发模型，系统被拆解为多个高效解耦的微组件。

### 2.1 核心组件职责
- **Session Manager**: 作为连接与状态的总指挥。管理会话生命周期，处理会话创建、过期清理、断线重连和心跳维护。具备线程安全的会话注册表（如 `sync.Map`）。
- **SSH Manager**: 负责与后端的真实服务器建立底层 SSH 连接（基于 `crypto/ssh` 包）。负责安全协商、密钥管理和 PTY 请求。
- **Multiexec Engine**: 多路输入广播引擎。处理异步输入、实现多节点独立 FIFO 写入保序、在遇到下游慢节点时执行精确限流。
- **Stream Router (输出网关)**: 20ms 窗口聚合路由。采用定时 Ticker 批量收取多 Session 输出，封装为自定义二进制帧，避免网络 IO 过载。
- **Diff Engine**: 实现输出隔离与内容差异识别。其完整处理流为：ANSI Strip -> Regex Filter -> Split -> Hash -> Compare。
- **Audit Service**: 旁路审计模块。异步且并发地记录命令（结构化脱敏）和流数据（Raw Stream）。使用流式压缩（Zstd）并持久化至分片对象存储（如 S3）。
- **Risk Control & Security**: 实时风控探针。拦截恶意命令，支持高危命令二次阻断或审批机制。
- **Resize Controller**: 监听前端传入的 `cols/rows`，并映射调用 SSH 的 `window-change`。
- **Flow Control (Backpressure)**: 基于水位的反压流控引擎。动态监测缓冲区大小。
- **Context Checker**: 上下文一致性检查模块。利用发送 `\x05` 或 `\n` 探测目标状态，识别 shell/tui。
- **Resource Guard**: 全局资源看门狗。强管控系统的 Goroutine 数量、文件描述符（FD）、连接速率。

### 2.2 线程安全与并发设计
- **无锁化优先**: 尽量采用 Channel 进行跨模块的数据通讯，而非共享内存和 Mutex。
- **连接池管理**: 对外的 SSH 连接复用 TCP 连接，并在逻辑层实现 PTY multiplexing 以提高效率。
- **安全下线**: 当出现 `panic` 或不可恢复的错误时，系统应当保证 `defer` 回调能正确释放 fd、关闭 socket，以防资源泄露。

## 3. WebSocket 优化与通信工程
- 强制使用 **Binary Frame** 进行数据传输。
- 利用多路复用设计，将 N 个终端的数据汇聚于 1 个 TCP (WebSocket) 物理通道上。
- **内存优化**: 采用 Go `sync.Pool` 预分配与复用二进制 Buffer 和各种短期对象，极大降低 GC 压力，减少长亭顿（STW）。
