# 性能与容量设计

本系统在设计时需要处理大规模长连接、高频事件分发以及高并发的流数据交互。为保障系统高可用，我们制定了以下严格的性能优化、容量配额和防线控制策略。

## 1. 系统级资源与容量配额

### 1.1 用户级配额
- **最大会话限制**:
  - 单用户同时开启的活跃 Session 数量建议硬性限制为 **≤ 20** 个。
  - 对于批量执行（Multiexec），一次性广播的目标终端数量不应超过该值，超过该阈值时 UI 将阻止操作并给出警告。
- **并发连接数**: 单用户的物理 WebSocket 连接一般限定在 1-3 个以内。

### 1.2 全局系统配额
- **SSH 连接数**: 全局 SSH 连接数（包含 PTY 请求）根据 Gateway 的宿主机资源进行限制。例如 4C8G 节点可承受 1000~2000 个活跃 SSH 连接。
- **文件描述符保护 (FD Guard)**:
  - 系统启动时必须执行强制预检：要求系统 `ulimit -n ≥ 4096`，否则拒绝启动。
  - 运行时加入文件描述符的动态监测，当系统剩余 FD 占用达到 85% 时，停止接收新的 Session 连接请求。
- **僵尸连接快速释放**:
  - 通过心跳机制（Ping/Pong），超过 15 秒无响应即断开 WebSocket。
  - 当 WebSocket 断开或 SSH 会话结束时，系统将立即且彻底地释放相关 Goroutine、PTY 通道及底层 socket，防止内存泄露。

## 2. 流控与反压（Backpressure）机制

为了防止慢客户端导致服务端内存暴涨（OOM），或远端服务器快速疯狂输出导致浏览器 WebGL/Canvas 卡死甚至崩溃，需采取双向防线：

### 2.1 后端流控 (Server-Side)
- **队列缓冲区**: 每个 session 分配独立的消息写缓冲队列（Channel 或 RingBuffer），默认上限 1MB 或 1024 个数据包。
- **慢节点熔断 (反压)**: 当远端不断产生海量日志（例如执行了 `cat /var/log/messages` 或无限死循环输出），而目标前端消费速度慢（因网络不佳或 CPU 算力不足）使得缓冲持续增长达到上限水位（例如 85%）时：
  - 系统立即标记该节点为 `isSlow = true`。
  - **暂停读取**: 后端立刻暂停执行从该 Session 的 SSH PTY 读取输出的操作。这会触发底层 TCP 连接窗口变小，进而反向将压力传递（反压）至目标机器的 SSHD 服务进程，迫使远端暂停吐出数据。
  - **恢复读取**: 等待客户端逐渐消费完内存积压消息，直至水位线回落到安全阈值（如 50%）后，再解开阻塞，恢复正常的读取分发。
- **聚合下发**: 配合 Stream Router 的 20ms 定时器窗口，合并碎包，降低发送频率和 TCP 包开销。

### 2.2 前端流控 (Client-Side)
- **渲染限制**:
  - 对 xterm 的 `Scrollback`（回滚历史）行数进行强限制，建议配置为 `5000` 行。若不设限制，长时间挂机的终端将积累几十万行 DOM/内存节点，直接导致浏览器崩溃（OOM）。
  - 在输出极快时，前端采取帧率节流策略。利用 `requestAnimationFrame` 将多次接收到的流数据进行合并，然后统一批量调用 xterm 的 `write` 方法，以保证浏览器 UI 主线程的流畅度和对用户的响应性。

## 3. 网络与传输层优化

- **二进制封包**: 彻底废弃 JSON 传输结构，全面拥抱基于二进制协议的数据封装，压缩体积并消除 String 转换开销。
- **多路复用 (Multiplexing)**: 在一条 WebSocket 长连接上复用用户所有的终端流数据，通过 SessionId 实现虚拟子通道。

## 4. 内存与 GC 优化

- **分级缓冲池 (Tiered Buffer Pool)**: 在高频的二进制帧读写流转路径上，绝对禁止直接使用 `make([]byte, size)` 动态分配堆内存切片。系统利用 Go 语言的 `sync.Pool` 预先建立对象池。为了防止不同大小的数据包造成内存碎片和浪费，必须实现分级缓冲池：维护针对小控制包（如 `256B`）、中等普通包（如 `4KB`）和极限聚合包（如 `32KB`）等不同规格的 Buffer 队列。
- **防止 STW 抖动**: 每次从 SSH 通道读出数据或向 WebSocket 封包前，从匹配的 Pool 中借出（`Get()`）合适大小的 Buffer，在数据发送完毕后，清空数据（`Reset()`）并立即归还（`Put()`）回池中。此举将原本每秒数万次的生命周期极短的内存分配操作，转化为极低成本的指针移动操作，从根本上削除了大量垃圾对象的生成。这极大降低了 Go 运行时的垃圾回收（GC）标记与清扫压力，成功避免了大流量洪峰下的 STW（Stop-The-World）导致的全局延迟抖动。

## 5. 可观测性与监控告警 (Observability & Metrics)

为了保障系统在逼近容量上限时能被及时发现，系统必须暴露以下核心监控指标（如 Prometheus /metrics 端点）：

### 5.1 资源与容量指标
- `gateway_active_connections`: 当前 Gateway 承载的活跃 WebSocket 连接数。
- `gateway_ssh_sessions_total`: 当前系统维持的底层 SSH 隧道总数（告警阈值：超过单机配额的 80%）。
- `gateway_fd_usage_percent`: 当前进程文件描述符 (FD) 占用百分比。依据 FD Guard 机制，当到达 85% 时应触发 P1 级别告警。

### 5.2 性能与流控指标
- `gateway_buffer_pool_hit_rate`: `sync.Pool` 二进制缓冲池的命中率，衡量内存复用效率。
- `gateway_slow_nodes_total`: 当前被标记为 `isSlow = true` 且正在触发反压暂停读取的慢节点数量。
- `gateway_stream_router_latency_ms`: Stream Router 20ms 定时器的实际调度延迟，若 CPU 过载导致延迟抖动，该值会显著飙升。
