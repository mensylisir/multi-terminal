# 性能与容量设计

本系统在设计时需要处理大规模长连接、高频事件分发以及高并发的流数据交互。为保障系统高可用，我们制定了以下严格的性能优化、容量配额和防线控制策略。

## 1. 系统级资源与容量配额

### 1.1 用户级配额
- **最大会话限制**:
  - 单用户同时开启的活跃 Session 数量建议硬性限制为 **≤ 20** 个。
  - 对于批量执行（Multiexec），一次性广播的目标终端数量不应超过该值，超过该阈值时 UI 将阻止操作并给出警告。
- **并发连接数**: 单用户的物理 WebSocket 连接一般限定在 1-3 个以内。

### 1.2 全局系统配额
- **SSH 连接数**: 全局 SSH 连接数（包含 PTY 请求）根据 Gateway 的宿主机资源进行限制。例如 4C8G 节点可承受 1000~2000 个活跃 SSH 连接。
- **文件描述符保护 (FD Guard)**:
  - 系统启动时必须执行强制预检：要求系统 `ulimit -n ≥ 4096`，否则拒绝启动。
  - 运行时加入文件描述符的动态监测，当系统剩余 FD 占用达到 85% 时，停止接收新的 Session 连接请求。
- **僵尸连接快速释放**:
  - 通过心跳机制（Ping/Pong），超过 15 秒无响应即断开 WebSocket。
  - 当 WebSocket 断开或 SSH 会话结束时，系统将立即且彻底地释放相关 Goroutine、PTY 通道及底层 socket，防止内存泄露。

## 2. 流控与反压（Backpressure）机制

为了防止慢客户端导致服务端内存暴涨（OOM），或远端服务器快速疯狂输出导致浏览器 WebGL/Canvas 卡死甚至崩溃，需采取双向防线：

### 2.1 后端流控 (Server-Side)
- **队列缓冲区**: 每个 session 分配独立的消息写缓冲队列（Channel 或 RingBuffer），默认上限 1MB 或 1024 个数据包。
- **慢节点熔断**: 当远端不断产生日志，而目标前端消费过慢使得缓冲持续增长达到上限水位时：
  - 立即标记该节点为 `isSlow = true`。
  - **暂停读取**: 后端立刻停止从该 Session 的 SSH PTY 读取输出（触发 TCP 反压至目标机）。
  - 等待客户端消费完积压消息，降低到安全水位（如 50%）后再恢复读取。
- **聚合下发**: 配合 Stream Router 的 20ms 定时器窗口，合并碎包，降低帧率。

### 2.2 前端流控 (Client-Side)
- **渲染限制**:
  - 对 xterm 的 `Scrollback`（回滚历史）行数进行强限制，建议配置为 `5000` 行。
  - 在输出极快时，前端采取帧率节流（RequestAnimationFrame），保证浏览器 UI 主线程的流畅度和响应性。

## 3. 网络与传输层优化

- **二进制封包**: 彻底废弃 JSON 传输结构，全面拥抱基于二进制协议的数据封装，压缩体积并消除 String 转换开销。
- **多路复用 (Multiplexing)**: 在一条 WebSocket 长连接上复用用户所有的终端流数据，通过 SessionId 实现虚拟子通道。

## 4. 内存与 GC 优化

- **Go sync.Pool**: 后端核心流转路径上禁用直接分配切片。使用 Go 语言的 `sync.Pool` 建立对象池，大量复用不同规格（如 256B, 4KB, 32KB）的二进制 Byte 缓冲区。
- 此举能将大量生命周期极短的包级内存分配转化为池化复用，极大削弱垃圾回收（GC）开销，防止大流量下的 STW（Stop-The-World）导致的延迟抖动。
