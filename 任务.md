# 开发任务拆分列表 (任务.md)

基于系统架构与设计文档，本平台按阶段拆解为以下开发任务，用于指导后续编码实现。

## 阶段一：基础设施与核心通信 (Phase 1)
此阶段构建系统骨架，打通 WebSocket 通信与基本的前后端交互能力。

- [ ] **任务 1.1: Gateway 项目初始化与配置管理**
  - [ ] 1.1.1 搭建 Go 项目脚手架，设计标准目录结构（cmd, internal, pkg）。
  - [ ] 1.1.2 引入 Viper 或类似库配置环境变量解析（端口、日志级别、Redis 地址等）。
  - [ ] 1.1.3 实现基础的优雅停机（Graceful Shutdown）框架，监听 SIGUSR2 信号。
- [ ] **任务 1.2: WebSocket Server 基础实现**
  - [ ] 1.2.1 基于 Gorilla WebSocket 编写连接 Upgrade 与接管逻辑。
  - [ ] 1.2.2 定义并实现底层的心跳机制 (Ping/Pong)。
  - [ ] 1.2.3 实现超时自动断开及连接清理逻辑。
- [ ] **任务 1.3: 二进制协议编解码器开发**
  - [ ] 1.3.1 定义基于 Big-Endian 的 `FrameType`, `SessionCount`, `SessionId`, `Length`, `Data` 结构体。
  - [ ] 1.3.2 编写编码器（Serialize）与解码器（Deserialize）函数，包含数据截断和未知帧类型的边界错误处理。
  - [ ] 1.3.3 使用 `sync.Pool` 实现 256B, 4KB, 32KB 等不同容量 Buffer 的复用池。
- [ ] **任务 1.4: 前端核心通信层与 Web Worker 初始化**
  - [ ] 1.4.1 创建 Vue/React 前端项目脚手架。
  - [ ] 1.4.2 编写 Web Worker 用于收发 WebSocket 二进制数据，避免阻塞主线程。
  - [ ] 1.4.3 在 Worker 中使用 `DataView/Uint8Array` 实现解包拆流逻辑。
  - [ ] 1.4.4 实现主线程与 Worker 之间高频的 `postMessage` 消息分发机制。

## 阶段二：会话管理与终端接入 (Phase 2)
此阶段完成单机 SSH 连接与 xterm.js 界面呈现。

- [ ] **任务 2.1: Session Manager 模块**
  - [ ] 2.1.1 实现全局线程安全的 `sync.Map` 注册表，管理 Session 的创建、查询、销毁。
  - [ ] 2.1.2 定义 Session 状态机模型与流转逻辑（active, detached, expired, closed）。
  - [ ] 2.1.3 实现 Session 生命周期定时巡检（如清理 expired 会话）。
- [ ] **任务 2.2: SSH Manager 模块**
  - [ ] 2.2.1 封装 `crypto/ssh`，实现客户端密钥鉴权连接。
  - [ ] 2.2.2 申请远端 PTY 通道，暴露独立的 Read/Write 接口供上层调用。
  - [ ] 2.2.3 实现 SSH 连接的保活（Keepalive）探测逻辑。
- [ ] **任务 2.3: 前端 xterm.js 渲染接入**
  - [ ] 2.3.1 集成 `xterm.js` 核心库。
  - [ ] 2.3.2 引入并激活 `xterm-addon-webgl` 插件以开启 GPU 渲染。
  - [ ] 2.3.3 将 Worker 解析出的流数据驱动给 xterm 实例，并将键盘输入回传给 Worker 封包。
- [ ] **任务 2.4: 终端窗口自适应控制**
  - [ ] 2.4.1 监听前端浏览器 Window Resize 及 xterm 容器变化事件，计算并传递 Cols/Rows 帧。
  - [ ] 2.4.2 后端接收 Resize 控制帧，调用 SSH `window-change` 更新远端 PTY 大小。

## 阶段三：聚合路由与流控缓冲 (Phase 3)
解决高并发下的小包碎片化和内存积压问题。

- [ ] **任务 3.1: 终端输出缓冲队列**
  - [ ] 3.1.1 为每个 Session 建立带上限（如 1MB）的并发安全 RingBuffer。
  - [ ] 3.1.2 暴露当前 Buffer 水位的读取接口，供 Backpressure 引擎调用。
- [ ] **任务 3.2: Stream Router (20ms 聚合引擎)**
  - [ ] 3.2.1 实现定时 Ticker，每 20ms 遍历活跃 Session 列表。
  - [ ] 3.2.2 收割所有带有数据的 Buffer，并将其打包拼接为一个大的 DataFrame。
  - [ ] 3.2.3 下发聚合包，并处理网络异常时的错误回滚。
- [ ] **任务 3.3: Backpressure 流控机制**
  - [ ] 3.3.1 监测 Buffer 水位，当水位超 85% 时标记 Session `isSlow = true`。
  - [ ] 3.3.2 发出控制信号，暂停从该 Session 的 SSH PTY 进行 Read，产生网络层反压。
  - [ ] 3.3.3 水位回落到 50% 后恢复 Read。
- [ ] **任务 3.4: 前端乐观回显与渲染节流**
  - [ ] 3.4.1 在 xterm 中硬编码 Scrollback 上限为 5000 行，防 OOM。
  - [ ] 3.4.2 利用 `requestAnimationFrame` 将多次收到的数据进行节流和批量 `write` 渲染。

## 阶段四：批量广播 (Multiexec) 与一致性保障 (Phase 4)
实现本系统的核心卖点：安全可控的多节点命令广播。

- [ ] **任务 4.1: 前端多终端布局与状态同步**
  - [ ] 4.1.1 引入 Zustand/Pinia 管理 `activeSessionId` 及 `selectedSessionIds` 状态机。
  - [ ] 4.1.2 支持快捷键（如 Shift+Click）多选，以及多终端的视图排版切换。
- [ ] **任务 4.2: Multiexec Engine 广播逻辑**
  - [ ] 4.2.1 拦截 active 终端的用户输入，将其克隆并投递至所有选定目标的异步写入通道。
  - [ ] 4.2.2 实现严格的 FIFO 保序逻辑，并捕获慢节点的缓冲满错误以生成警告。
- [ ] **任务 4.3: 上下文状态探测 (Context Checker)**
  - [ ] 4.3.1 在广播执行前，系统隐蔽发送探针（如 ``）。
  - [ ] 4.3.2 接收反馈并在 2 秒超时内识别终端是处于 `shell` 还是全屏的 `tui`。
  - [ ] 4.3.3 将处于 `tui` 和无响应的节点自动移出广播目标，并在前端覆盖红色锁定遮罩。
- [ ] **任务 4.4: 终端回显锁定器**
  - [ ] 4.4.1 在前端实现 Multiexec 时，active 终端执行本地“乐观回显”。
  - [ ] 4.4.2 临时锁定后端所有的回显输出帧的渲染逻辑。
  - [ ] 4.4.3 等待短暂延迟后解锁，防止多终端同时输出造成重影。

## 阶段五：数据隔离审计与高可用 (Phase 5)
实现高阶特性：审计日志、差异比对及优雅升级。

- [ ] **任务 5.1: Diff Engine (输出差异识别)**
  - [ ] 5.1.1 编写高效的正则表达式引擎，剥离 `Raw Output` 中的 ANSI 颜色和终端定位控制符。
  - [ ] 5.1.2 引入 `Regex Filter` 逻辑，过滤掉命令行中动态输出的进度条或无用的时钟信息。
  - [ ] 5.1.3 按照行 (`\n`) 切分清洗后的纯净字符串，并计算其 MD5 或 SHA1 的 Hash 值。
  - [ ] 5.1.4 将各节点的最终 Hash 值与远端命令执行的 `Exit Code` 进行比对，将结果集打包推送给前端以展示多机异同高亮。
- [ ] **任务 5.2: Audit Service (高保真审计)**
  - [ ] 5.2.1 捕获 `Raw Input` 和 `Raw Output` 流数据，并打上 `Timestamp` 和 `SessionId` 标签。
  - [ ] 5.2.2 组装结构化日志 (JSON 格式)，并针对敏感字段执行拦截与脱敏（如隐藏 Password 输入）。
  - [ ] 5.2.3 引入 `zstd` 库，对产生的日志数据执行流式高压缩率压缩。
  - [ ] 5.2.4 将压缩后的审计分片定时或按体积持久化上传至云端对象存储 (S3/Minio)。
- [ ] **任务 5.3: 断线重连与状态恢复**
  - [ ] 5.3.1 前端 WebSocket 断开后不销毁 SSH，Session 转为 `detached` 状态。
  - [ ] 5.3.2 重新发起连接并携带原 `SessionId` 握手。
  - [ ] 5.3.3 重建连接后，将后台积压的 Buffer 重新分发回放给前端。
- [ ] **任务 5.4: 平滑升级 (Graceful Restart) 支持**
  - [ ] 5.4.1 网关收到退出信号后，阻断新连接进入。
  - [ ] 5.4.2 挂起当前流并将活跃 Session 元数据序列化存入 Redis。
  - [ ] 5.4.3 通过 Unix Domain Socket 转移底层的 TCP FD 到刚拉起的新进程，新进程无感知接管。

## 阶段六：测试与持续集成 (Phase 6)
保障系统质量与交付标准，实现准生产级交付。

- [ ] **任务 6.1: 单元与集成测试框架**
  - [ ] 6.1.1 编写 Go 后端核心逻辑（二进制协议编解码、流控算法）的单元测试。
  - [ ] 6.1.2 编写前端 Web Worker 消息解析的单元测试。
  - [ ] 6.1.3 搭建基于本地 Docker 的 SSH 桩服务，进行自动化的前后端集成联调测试。
- [ ] **任务 6.2: 端到端 (E2E) 自动化测试**
  - [ ] 6.2.1 编写脚本模拟真实的浏览器操作，测试 Multiexec 多选输入是否同步。
  - [ ] 6.2.2 模拟异常网络断开与恢复场景，断言终端界面的回放恢复是否符合预期。
- [ ] **任务 6.3: CI/CD 与部署流水线**
  - [ ] 6.3.1 编写项目通用的 `Dockerfile`，将静态前端资源注入 Go 二进制。
  - [ ] 6.3.2 编写 CI 配置文件（如 GitHub Actions 或 GitLab CI），实现代码 Push 自动运行 Linter 与 Tests。
  - [ ] 6.3.3 制定自动化发版与基于负载均衡器平滑升级的发布脚本策略。

## 阶段七：可观测性与监控告警 (Phase 7)
基于暴露的核心性能指标建立自动化监控与防御预警体系。

- [ ] **任务 7.1: 指标暴露与埋点 (Prometheus Metrics)**
  - [ ] 7.1.1 暴露 `gateway_active_connections` 和 `gateway_ssh_sessions_total` 等连接容量指标。
  - [ ] 7.1.2 暴露 `gateway_fd_usage_percent` 文件描述符水位指标，以监控系统极限。
  - [ ] 7.1.3 暴露 `gateway_buffer_pool_hit_rate` 和 `gateway_slow_nodes_total` 以评估 GC 压力及慢节点熔断状态。
- [ ] **任务 7.2: 可视化与告警策略**
  - [ ] 7.2.1 搭建 Grafana Dashboard，实时展示 Gateway 节点集群的并发状态。
  - [ ] 7.2.2 配置 Alertmanager 告警规则（如 FD 占用超 85% 触发 P1 报警，SSH 会话逼近单机上限触发扩容警告）。

## 阶段八：安全风控与命令拦截 (Phase 8)
保障生产环境系统的核心防线，落地命令预检及拦截。

- [ ] **任务 8.1: 规则引擎基础建设**
  - [ ] 8.1.1 定义 JSON 格式的规则引擎配置文件结构（包含级别、动作、正则模式等）。
  - [ ] 8.1.2 编写后端 `Risk Control` 微件，实现在内存中缓存规则并在初始化时预编译 Go `regexp` 实例。
- [ ] **任务 8.2: 实时输入探测与拦截**
  - [ ] 8.2.1 拦截并组装前端传来的 `Raw Input` 字符碎片（处理退格等控制逻辑），还原为 `CommandContext`。
  - [ ] 8.2.2 在将命令实际送入 SSH 隧道前进行正则匹配（Pre-Execution Check）。
  - [ ] 8.2.3 当匹配到 `High` 级别指令时，切断后续输入并在终端打印红色拦截警告；记录安全审计告警。
- [ ] **任务 8.3: 前端二次确认机制 (Medium Level)**
  - [ ] 8.3.1 当后端命中 `Medium` 级别规则（如 `reboot`）时，下发控制帧暂停执行，并向前端发回挂起信号。
  - [ ] 8.3.2 前端全局状态收到挂起信号，弹出包含警告信息的确认弹窗（Confirm Modal）。
  - [ ] 8.3.3 用户点击确认后，发送包含特殊验证标志的帧让后端放行执行，否则清理命令缓冲。
