# 测试方案 (Test Strategy)

## 1. 总体测试目标与架构规范

本测试方案旨在为“多主机实时终端系统”提供全面的质量保障策略。该系统基于 Go Gateway 与 xterm.js WebGL，并引入了专有的二进制复用协议（多路复用）。核心目标是验证整个架构流向上的**每一个节点、每一次心跳（Heartbeat）、以及在极端并发或恶意输入条件下的防线（Risk Control）是否绝对可靠。**

### 1.1 架构规范与边界
- **协议栈底座**：系统内部所有通信强依赖自定义二进制 Frame：
  - `FrameTypeHeartbeat (0x01)`
  - `FrameTypeTerminalOutput (0x02)`
  - `FrameTypeTerminalInput (0x03)`
  - `FrameTypeWindowResize (0x04)`
  - `FrameTypeSessionControl (0x05)`
- **核心组件交互边界**：
  - Frontend (WebWorker) <-> WSS (Gateway Router) 之间的二进制编解码。
  - Gateway (Session Manager) <-> S3 (审计存储) / Redis (锁及状态管理)。
  - Gateway (SSH Manager) <-> 目标主机系统的 PTY 流交换。

## 2. 深入测试范围 (Test Scope & Depth)

### 2.1 协议层深度验证 (Protocol & Codec)
- **帧解析边界 (ErrBufferTooSmall / ErrInvalidFrame)**：故意构造断层、粘包、或者长度字段（Length）与 Payload Data 实际字节数不匹配的畸形数据包，验证 Gateway Router 能够直接丢弃畸形包而不断开主 WebSocket 连接或发生 Panic。
- **心跳机制与会话续命**：验证前端持续发送 `FrameTypeHeartbeat (0x01)` 维持 `LastActiveAt` 状态，并且断网 15 秒内心跳重连（Session 恢复），超过 15 秒网关立即清除 Goroutine、释放 PTY。

### 2.2 极端并发性能与系统防爆 (Performance & Throttling)
- **Goroutine 防爆与内存压测**：在 2000 个活跃长连接（单个连接 20 个 Multiexec 并发，即 40,000 个目标主机会话）并发场景下，使用 pprof 观测 Gateway 内部 Router 的 `sync.Pool` 借还情况，防止 GC OOM。
- **FD 预检与动态卡点机制**：触发网关层预检，验证如果系统级 `ulimit -n < 4096` 或达到 85% 水位时，新的会话创建请求被明确拒绝返回错误并触发告警。
- **流控反压防线 (Backpressure)**：针对 `isSlow = true` 的熔断机制测试。当后端缓冲队列积压（如执行高并发 `cat` 几十个GB日志），Gateway 立即停止从远端 SSH PTY 调用 `Read` 接口，等待前端确认后再恢复。

### 2.3 深度功能与异常分支 (Functional & Edge Branches)
- **Multiexec (批量分发引擎)**：多路并发输入时，对包含控制符 `\x03` (Ctrl+C), `\x04` (Ctrl+D), `\x1b` (Escape ANSI 序列) 进行保序分发。
- **Risk Control (智能防误与拦截)**：
  - 拦截预检机制（PromptState）：发送探针前主机已在 `vim` / `top` / `mysql` 等全屏或交互 TUI 中。
  - 指令正则预检：执行包含高危模式的指令（如 `rm -rf /`，`chmod 777`）触发阻断逻辑（OP_BLOCK）。

### 2.4 可靠性与兼容性分支 (Reliability)
- **优雅重启 (Graceful Restart) - FD Handoff**：升级 Gateway 版本时，底层 FD 通过 unix socket 传递，保持现存的所有 Frontend 页面 WSS 连接和后台 SSH 均不断开。

## 3. 测试环境拓扑与工具要求

- **底层环境**：要求配置解除全部限制 (`ulimit -n 65535`)，必须部署 Redis Cluster (处理 SessionMeta锁) 和 MinIO (作为 S3 审计替代验证)。
- **网络模拟 (Chaos Mesh)**：用于模拟高延迟 (300ms) 及丢包环境 (TCP Drop)，验证 Gateway 的读写缓冲限额和客户端重传机制。
- **测试工具链**：禁止直接使用单纯的 cURL 进行功能测试，必须编写专有工具构造 `Frame` 二进制封包与 Gateway 进行 WSS 握手和 `0x03` / `0x02` Frame 通信。
